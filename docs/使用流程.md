# ePgdb 使用流程

本文档按当前仓库代码整理，覆盖 schema 定义、静态生成、服务启动、表结构同步、日常 CRUD 和性能测试入口。

## 1. 准备 schema 模块

当前项目推荐把 schema 写在 `src/schema` 目录下，每个 `*_schema.erl` 模块导出若干个零参函数，每个函数代表一张表，并返回
`#schema{}`。

示例：

```erlang
-module(pg_player_schema).
-include("pgdbSchema.hrl").
-compile([export_all, nowarn_export_all]).

players() ->
    #schema{
        repr = record,
        comment = "玩家主表",
        fields = [
            #schField{name = id, dbType = ?pg_bigserial, opts = [?pg_primary_key]},
            #schField{name = name, dbType = ?pg_varchar(64), opts = [?pg_not_null, ?pg_unique]},
            #schField{name = level, dbType = ?pg_integer, default = 1},
            #schField{name = gold, dbType = ?pg_bigint, default = 0},
            #schField{name = profile, dbType = ?pg_jsonb, default = #{}, codec = ?codec_json, erlType = "map()"}
        ]
    }.
```

当前静态生成器只扫描零参导出函数，并要求函数返回 `#schema{}`，因此 schema 源文件应统一使用这种写法。

## 2. 生成静态 schema 模块

写完 schema 后，执行：

```erlang
genPSchema:gen().
```

默认等价于：

```erlang
genPSchema:gen("./src/schema", dbSchemaDef, "./src/schema", "./include", "./include").
```

生成结果包括：

- `src/schema/dbSchemaDef.erl`：统一的静态 schema 查询模块
- `include/*_schema.hrl`：各 schema 模块对应的 record/type 头文件

运行时的 insert/select/update/delete/upsert、where 编码、结果解码，都会优先走这份静态 schema。

## 3. 编译项目

```bash
rebar3 compile
```

如果 schema 有改动，推荐先重新执行一次 `genPSchema:gen()`，再编译。

## 4. 启动 ePgdb

当前主入口是 `ePgdb:start/6`：

```erlang
{ok, _Pid} = ePgdb:start("127.0.0.1", 5432, "postgres", "postgres", "game_db", [
    {wFCnt, 10},
    {wKeepTime, 1800000},
    {slowThreshold, 1000},
    {heartbeatInterval, 30000},
    {wArgs, [
        {timeout, 5000},
        {application_name, "game_server"},
        {tcp_opts, [{keepalive, true}, {nodelay, true}]}
    ]}
]).
```

常用参数说明：

- `wFCnt`：常驻 worker 数，也就是常驻数据库连接数
- `wKeepTime`：空闲连接在池内保留多久，超过后会被池管理器回收
- `slowThreshold`：慢查询告警阈值，单位毫秒
- `heartbeatInterval`：ePgdb 额外提供的空闲连接心跳周期，单位毫秒；大于 0 时会定期对 idle 连接执行 `SELECT 1`
- `wArgs`：传给 `pgdbWorker` 的连接参数和扩展项
- `wArgs` 中的 `timeout`：epgsql 连接超时，默认 5000ms
- `wArgs` 中的 `tcpOpts` / `tcp_opts`：TCP 选项，传给 epgsql 的 `tcp_opts`
- `wArgs` 中的 `ssl` / `sslOpts` / `ssl_opts`：SSL 开关和 SSL 参数
- `wArgs` 中还支持 epgsql 其它连接参数，例如 `application_name`、`socket_active`、`async`、`codecs`、`nulls`

更完整的 epgsql 连接参数说明见 `docs/epgsql-connect-options.md`。

## 5. 启动时自动做的事情

`ePgdb:start/6` 当前会按下面顺序执行：

1. 启动 ePgdb 及依赖应用
2. 先连接 PostgreSQL 的 `postgres` 维护库，必要时自动创建目标数据库
3. 打开 eFaw 工厂，创建 `pgdbWorker`
4. 调用 `syncCheckSchema/0`，按 `dbSchemaDef:getTables()` 中的表定义自动同步结构

当前自动同步主要做的是：

- 表不存在时自动建表
- 字段缺失时自动补齐
- 字段类型不一致时尝试 `alterColumnType`

这意味着开发期最常见的工作流是：改 schema、重新生成 `dbSchemaDef`、重启服务或显式调用 `syncSchema/1`。

## 6. 日常业务读写

### 6.1 插入

```erlang
ok = ePgdb:insert(#{
    table_name => players,
    id => 1,
    name => <<"Alice">>,
    level => 12,
    gold => 8800,
    profile => #{rank => 3, server => <<"s1">>}
}).
```

### 6.2 按主键读取

```erlang
{ok, [Player]} = ePgdb:get(players, #{id => 1}).
```

### 6.3 条件查询

```erlang
{ok, Rows} = ePgdb:select(players, #{level => {'>=', 10}, vip => true}, [
    {fields, [id, name, level, profile]},
    {order_by, [{level, desc}, {id, asc}]},
    {limit, 100}
]).
```

### 6.4 更新

```erlang
{ok, 1} = ePgdb:update(#{
    table_name => players,
    gold => 9900,
    profile => #{rank => 4, server => <<"s1">>}
}, [gold, profile], #{id => 1}).
```

### 6.5 删除

```erlang
{ok, 1} = ePgdb:delete(players, #{id => 1}).
```

### 6.6 事务

```erlang
{ok, ok} = ePgdb:transaction(fun(Conn) ->
    ok = ePgdb:insert(Conn, #{
        table_name => bench_kv,
        key => <<"k1">>,
        value => #{n => 1},
        version => 1,
        ttl => 0
    }),
    {ok, 1} = ePgdb:update(Conn, #{
        table_name => players,
        gold => 100
    }, [gold], #{id => 1}),
    ok
end).
```

## 7. 结构变更时怎么做

如果你给表增加字段或调整字段类型，推荐流程是：

1. 修改 `src/schema/*_schema.erl`
2. 重新执行 `genPSchema:gen()`
3. 重新编译项目
4. 启动时自动同步，或显式执行 `ePgdb:syncSchema(Table)`

示例：

```erlang
ok = ePgdb:syncSchema(players).
```

## 8. 测试与性能测试

### 8.1 功能测试

```bash
rebar3 eunit --module=pgdb_admin_tests --verbose
rebar3 eunit --module=pgdb_crud_tests --verbose
rebar3 eunit --module=pgdb_select_tests --verbose
rebar3 eunit --module=pgdb_bench_tests --verbose
```

### 8.2 性能测试

进入 test shell：

```bash
rebar3 as test shell
```

跑整套压测：

```erlang
tcPgdb:main().
tcPgdb:main("", 4, 500).
tcPgdb:main("tc_pgdb_bench.log", 4, 500).
```

单独跑某一组：

```erlang
tcPgdb:crud_performance_test(1, 1000).
tcPgdb:batch_performance_test(1, 300).
tcPgdb:structure_performance_test(1, 300).
tcPgdb:query_performance_test(1, 200).
tcPgdb:transaction_performance_test(1, 300).
tcPgdb:mixed_workload_performance_test(4, 500).
```

参数含义：

- 第一个参数 `ProcessCnt`：并发进程数
- 第二个参数 `LoopCnt`：每个进程执行次数

开发机 quick 跑法建议：

- `tcPgdb:main("", 1, 100).`
- `tcPgdb:crud_performance_test(1, 200).`
- `tcPgdb:mixed_workload_performance_test(2, 100).`

单机正式压测建议：

- 先跑 `tcPgdb:crud_performance_test(1, 300)` 预热
- 再跑 `tcPgdb:main("single_baseline.log", 1, 1000)` 建单进程基线
- 然后逐步提高到 `4`、`8` 并发做对比

## 9. 当前项目里最重要的几个文件

- `src/ePgdb.erl`：主 API，包含 CRUD、DDL、事务、迁移、自省
- `src/pgdb/genPSchema.erl`：schema 静态代码生成器
- `src/schema/dbSchemaDef.erl`：生成后的静态 schema 模块
- `src/schema/*_schema.erl`：schema 源文件
- `include/*_schema.hrl`：生成的 record/type 头文件
- `test/src/tcPgdb.erl`：专门的性能压测模块
- `test/pgdb_*_tests.erl`：EUnit 功能测试与轻量基准测试

## 10. 推荐习惯

- schema 改完立刻重新生成 `dbSchemaDef`
- 业务层尽量统一走 ePgdb API，不要一半手写 SQL、一半走 schema codec
- 原生 SQL 保留给复杂报表、特殊优化点和迁移场景
- 做性能比较时固定 PostgreSQL 配置、数据量、连接池参数和机器负载，避免把环境抖动当成代码收益
